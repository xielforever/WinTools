# WinTools 设计文档

## 1. 文档目标

本文档用于说明 WinTools 当前版本的总体架构、模块边界、目录大小统计模块实现、线程模型与数据模型设计，作为后续扩展和重构的基线文档。

## 2. 技术栈与运行环境

- 语言：Python 3.13+
- GUI：tkinter + ttk
- 数据存储：SQLite（本地单文件）
- 平台：Windows

## 3. 总体架构

项目采用“主框架 + 模块插件”的结构：

- `main.py`：程序入口
- `wintools/app.py`：主窗口与全局布局、模块切换、状态栏
- `wintools/base.py`：模块接口契约（`BaseModule`）
- `wintools/module_registry.py`：模块注册中心
- `modules/<module_name>/`：每个功能模块独立目录

目录大小统计模块：

- `modules/dir_size/core.py`：目录扫描核心逻辑（无 UI）
- `modules/dir_size/ui.py`：模块 UI 与交互控制
- `modules/dir_size/storage.py`：SQLite 读写与查询

## 4. 模块化设计原则

### 4.1 BaseModule 契约

每个模块必须实现：

- `mount(parent, set_status)`：挂载模块界面
- `unmount()`：释放资源（停止线程、销毁菜单等）

### 4.2 解耦约束

- UI 与扫描逻辑分离：`ui.py` 不直接实现目录遍历算法
- UI 与 SQL 分离：`ui.py` 仅调用 storage 接口，不拼接 SQL
- 主窗口不感知模块内部业务细节，只负责挂载和状态转发

## 5. 目录大小统计模块设计

### 5.1 功能边界

- 仅展示当前层级：根目录 + 直接子目录
- 每一行的大小按递归总大小计算（全子树）
- 结果默认按大小降序
- 支持后台扫描、停止扫描
- 支持右键深入扫描、趋势分析、打开目录位置
- 每次扫描结果持久化到 SQLite

### 5.2 扫描流程

1. 用户选择目录并点击“开始扫描”
2. UI 创建结果标签页并显示“扫描中”
3. 后台线程执行 `scan_directory_sizes`
4. 扫描完成后写入 SQLite 快照
5. 主线程取回结果，刷新表格并更新状态栏

### 5.3 并发与 UI 响应

- 扫描在独立线程执行，避免阻塞 tkinter 主线程
- 线程间通过 `queue.Queue` 传递结果
- 主线程使用 `after()` 轮询队列
- 取消扫描通过 `threading.Event`（`cancel_event`）协作中断

## 6. 数据模型设计（SQLite）

数据库文件：`data/dir_size_history.db`

### 6.1 表结构

#### `scan_runs`

记录“每次扫描任务”的根目录快照。

- `id` INTEGER PK AUTOINCREMENT
- `scanned_at` TEXT NOT NULL（ISO 时间）
- `root_path` TEXT NOT NULL（绝对路径）
- `root_path_key` TEXT NOT NULL（规范化路径：lower + resolve）
- `root_size_bytes` INTEGER NOT NULL
- `dir_count` INTEGER NOT NULL（本次结果行数）

#### `scan_top_dirs`

记录“每次扫描结果中的每一行目录”。

- `run_id` INTEGER NOT NULL（FK -> `scan_runs.id`，级联删除）
- `path` TEXT NOT NULL（目录路径）
- `size_bytes` INTEGER NOT NULL
- `path_key` TEXT（规范化路径，用于趋势查询）

说明：虽保留历史命名 `scan_top_dirs`，当前语义为“当前层级全部结果行（根目录 + 直接子目录）”。

### 6.2 索引设计

- `idx_scan_runs_root_time(root_path_key, scanned_at)`
  用于快速读取某根目录历史记录。

- `idx_scan_top_dirs_path_key(path_key)`
  用于右键分析任意目录（含子目录）的历史趋势。

### 6.3 关系模型

- 一次扫描任务（`scan_runs`）对应多条结果行（`scan_top_dirs`）
- 关系：`scan_runs(1) -> scan_top_dirs(N)`

## 7. 趋势分析查询设计

### 7.1 查询目标

对任意目录路径 `P`，返回时间序列点：

- `scanned_at`
- `root_size_bytes`（对应时刻目录总大小）

### 7.2 查询来源

趋势数据由两部分联合：

1. `P` 作为扫描根目录时（来自 `scan_runs`）
2. `P` 作为父目录扫描中的子目录行时（来自 `scan_top_dirs` + `scan_runs`）

联合后按时间升序排序，保证 `A` 的子目录 `B` 在“仅扫描过 A”时也能分析 `B` 趋势。

## 8. 数据一致性与演进策略

### 8.1 事务一致性

- 每次写入在单连接事务中完成
- 若写入异常，整次扫描快照回滚，避免脏数据

### 8.2 架构演进

- `init_db()` 中进行最小化 schema 迁移
- 自动补齐新增列（如 `path_key`）
- 自动回填历史行缺失字段

### 8.3 历史 JSON 策略

- 已切换 SQLite 为唯一读写源
- 历史 JSON/JSONL 不再参与业务逻辑

## 9. 异常处理策略

- 路径不存在/非目录：提示“路径异常”
- 权限不足：提示“权限异常”
- 其他系统错误：提示“系统异常”
- 未知异常：统一兜底并提示错误信息

异常在 UI 线程弹窗显示，同时状态栏更新。

## 10. 可扩展性规划

### 10.1 新模块接入

1. 在 `modules/<new_module>/` 新建模块目录
2. 实现 `BaseModule` 接口
3. 在 `module_registry.py` 注册模块

### 10.2 趋势能力扩展建议

- 增加“目录增长率”查询视图
- 支持按天/周聚合趋势点
- 引入图表导出（PNG/CSV）
- 为趋势分析增加筛选（时间区间、最小变化量）

## 11. 非功能性约束

- 保持 UI 主线程流畅，不允许扫描阻塞界面
- 存储层接口稳定，避免 UI 直接依赖 SQL
- 中文文案统一、异常提示可理解
- 代码应具备类型标注和必要注释，便于长期维护
